# 自适应端云推理：决策记录

日期：2026-07-18

状态：已收敛，作为 `spec.md` 的设计输入

调研基线：`4ede5d1`

## 1. 原始目标

用户在 Android 手机上运行较小的本地模型，同时在个人电脑或服务器运行较大的模型。应用在用户显式开启后，依据隐私边界、任务需求、手机资源和远端状态，决定本次请求在本地还是远端执行，形成轻量、可控、可解释的边缘协同。

成功不等于“尽量多用远端”，而是同时满足：

- `LocalOnly` 数据永不因自动决策离开设备；隐私状态未知时按 `LocalOnly` 处理。
- 每次运行只选择一个推理位置，执行中不偷偷换后端或重复发送。
- 简单任务和远端不可用时优先保留本地体验；复杂任务或手机资源受限时可使用已授权、已验证的远端。
- 用户能选择“本地 / 自动 / 远程”，并能看到本次实际使用的位置和原因。
- 第二阶段将“手工填写一个兼容接口”升级为“手机与一个可信边缘节点配对并感知其能力/负载”。

## 2. 两阶段的本质关系

两阶段共享同一个核心抽象：**模型放置决策（model placement）**，但不是同一项工程。

| 阶段 | 已有输入 | 新增能力 | 刻意不做 |
|---|---|---|---|
| 第一阶段：自适应选择 | 现有远程模型配置、连通性、本地模型状态、手机资源、任务结构 | 纯本地、确定性的放置策略；run 内固定；解释与审计 | 电脑发现、自动配对、远端实时负载、跨端配置同步 |
| 第二阶段：可信边缘节点 | 第一阶段的放置策略 | 单节点配对、身份与授权、能力/负载状态、幂等请求和取消 | 多节点调度、公共云控制面、运行中迁移、KV/隐藏状态传输 |

因此，两阶段是**交付和依赖顺序**，不是“本地小模型先推理、远端大模型再推理”的两级流水线。一个 run 只在一个位置生成；第一阶段验证决策器是否有价值，第二阶段提升远端候选的可信度、可观测性和易用性。

## 3. 已评估方案

### 方案 A：把 `Auto` 直接加进 `InferenceMode`

放弃。当前 `InferenceMode` 同时被 UI、隐私、取消和工具续写当作“正在使用的后端”。若加入 `Auto`，运行时无法知道应该取消哪个 runtime，也容易让用户中途切换模式改变同一个 run 的目的地。

### 方案 B：保留手动模式，新增独立的放置策略

采用。拆成三个互不替代的概念：

- `InferencePreference`：用户的长期偏好，`Local / Auto / Remote`。
- `MessagePrivacy`：本次数据是否允许出端，`LocalOnly / RemoteEligible`；未知按 `LocalOnly`。
- `RunPlacement`：本次 run 实际执行位置，`Local / Remote`，一经选择即固定。

这使“用户想要什么”“数据允许去哪里”“这次实际上在哪里运行”能分别测试和审计。

### 方案 C：用小模型或关键词判断复杂度

放弃。新的模型路由会增加延迟和失败面，关键词规则又难以解释和覆盖多语言。第一阶段只使用已有的结构化信号：估算 token、图片/能力需求、reasoning 档位、是否进入多步工具流程，以及资源状态。复杂度只用于候选都合法时的取舍，不能越过隐私和能力门禁。

### 方案 D：远端失败后自动改用本地重跑

放弃。首次远程发送后再本地重跑会造成重复计费、重复工具副作用、两份不一致回复，也无法证明远端是否已经处理。第一阶段只允许在**发送前**根据探测结果选择候选；发送后失败则结束该 run，由用户重试并创建新 run。

### 方案 E：直接让手机读取 Ollama/vLLM/llama.cpp 的各自状态接口

仅作为第一阶段的人工配置，不作为第二阶段架构。三者的数据面都能提供 OpenAI 兼容能力，但健康、模型、队列和负载接口不统一。第二阶段增加一个很薄的 `solin-edge` 适配层，把不同后端归一成稳定的移动端控制协议，模型推理仍由现有服务负责。

### 方案 F：第二阶段一开始做局域网发现和多节点调度

放弃。发现协议、节点排序、故障转移、证书身份和 UI 都会成倍扩大状态空间。首版只允许一个已配对的活动节点；通过二维码或用户主动导入邀请完成配置。发现和多节点是有真实需求后再加的独立能力。

## 4. 确认的关键决策

1. 默认偏好仍为 `Local`；旧版本配置迁移后不得自动开启远端。
2. `Auto` 是显式 opt-in 的远端能力，启用前复用现有远程数据披露。
3. 决策顺序固定为：隐私分类 → 所需能力 → 用户偏好 → 候选健康 → 资源/复杂度取舍。
4. 隐私、能力和用户强制偏好是硬门禁；资源和复杂度只是 `Auto` 的软策略。
5. `LocalOnly` 或隐私未知只能本地执行；本地不可用则明确阻断，绝不静默远传。
6. 手动 `Local` 和手动 `Remote` 保留当前“强制选择”语义，不做跨后端 fallback。
7. `Auto` 中远端只有在已授权、配置合法、能力匹配且近期探测为 `Reachable` 时才是候选。
8. 放置在首次 serving-model 调用前只决定一次，并绑定 `runId`；工具续写、overflow 重试和 steer 都复用同一放置；本地辅助模型不在该目标内。
9. 运行中出现新的 `LocalOnly` 工具结果时，远端 run 停止并解释，不自动切本地继续。
10. 路由记录只保存枚举化信号、策略版本、目标和原因码，不保存 prompt、工具原始参数、API key、endpoint 或 IP。
11. 第一阶段不因选择远端而自动卸载本地模型；内存回收策略与路由策略分开演进。
12. 第二阶段继续使用 OpenAI 兼容的数据面，新增最小的 Solin 控制面，不创造新的推理协议。
13. 第二阶段基线是系统信任的 HTTPS、短期邀请、设备级 bearer token 和可撤销配对；不实现自定义应用层加密。
14. 第二阶段先支持一个活动节点，不做自动发现、relay 或多节点负载均衡。
15. 请求使用稳定 `request_id`，edge 对收到的 body 自算 hash 并原子登记以实现幂等；SSE 必须显式完成，截断流不得触发工具执行。

## 5. 路由原则

优先级从高到低：

1. 隐私与本地证据边界。
2. 文本、视觉、工具调用、上下文长度等能力匹配。
3. 用户显式强制选择。
4. 候选是否就绪、已授权、可达且状态未过期。
5. Android `lowMemory`、Hot 和严重热状态等强远端偏置；只有 runtime admission 失败或 Emergency/Shutdown thermal 才是硬阻断。
6. 上下文压力、reasoning 档位、多步任务等复杂度信号。
7. 图片留在本地、简单任务低延迟等偏好。
8. 信号未知时回到本地；两边都不合法时阻断。

## 6. 第二阶段的信任边界

- 手机仍是策略、隐私分类、工具注册、安全确认和审计的权威。
- 边缘节点只接收 `RemoteEligible` 的模型输入并返回模型流；它不能直接调用手机工具。
- 模型服务和 `solin-edge` 运行在用户控制的电脑/服务器；HTTPS 终止点必须属于该信任边界。
- TLS 保护传输，但推理节点必然能看到明文 prompt。若未来引入不可信 relay，才需要基于成熟标准另行设计端到端加密。
- 二维码不是永久凭证，只含短期、单次邀请；正式 token 按设备签发、可单独撤销。

## 7. 明确不在本方案内

- 本地草稿 + 远端校正、speculative decoding 或模型集成。
- 跨端传输 KV cache、隐藏状态、会话数据库、记忆库或完整配置同步。
- 运行中的模型迁移、token 级调度或远端失败后的自动重放。
- 多手机、多活动节点、节点自动发现、云端账户/控制面、NAT relay。
- 自研证书信任管理、自签名证书信任器或自定义密码协议。
- 动态学习的路由权重、LLM 路由器或 prompt 关键词分类器。
- 直接在手机暴露 Prometheus 指标或把 provider 私有状态模型带入 Android。

## 8. 主要风险及约束

| 风险 | 约束 |
|---|---|
| 自动路由突破隐私边界 | 隐私先于路由；未知 fail closed；LocalOnly 远端调用数必须为 0 |
| UI 模式变化导致同一 run 换后端 | 放置绑定 `runId`，续写和取消不再读取可变偏好 |
| 远端状态陈旧导致错误选择 | 状态带服务端时间、TTL 和 boot ID；过期即不进入 Auto 候选 |
| 网络截断触发不完整工具调用 | 只有收到协议完成信号后才提交完整输出和工具请求 |
| 重试造成重复推理/副作用 | request ID + body hash；跨目标不自动重试；工具仍在手机确认后执行 |
| 配对邀请或 token 泄漏 | 短期单次邀请、设备级 token、加密存储、服务端哈希、撤销能力 |
| 为多 provider 引入过重网关 | `solin-edge` 只做认证、状态归一、幂等/取消和代理，不承载模型调度平台 |
| 远端选中但本地模型仍占内存 | 首版接受，单独度量；不把卸载与路由绑定 |

## 9. 待规格后验证的产品假设

这些不是未决接口，而是需要通过分阶段发布验证的假设：

- 只用结构化复杂度和资源档位，是否已经能显著改善复杂任务成功率和手机热/内存压力。
- 单个已配对节点是否覆盖个人边缘协同的主要场景。
- 用户是否需要看到简短原因，还是还需要高级诊断页。
- 远端选中时不卸载本地模型的内存收益是否足够；若不足，再单独设计生命周期策略。
