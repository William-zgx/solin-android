# 在手机上跑大模型：栖知 Solin 的端侧模型能力与应用场景

> 面向对端侧 AI / Agent 感兴趣的技术同学 · 概览介绍 · 约 5-8 分钟阅读

---

## 一、为什么要在手机上跑大模型？

过去一年，Agent 产品大多架在云端大模型之上。体验很好，但有三个绕不开的问题：

- **隐私**：聊天记录、屏幕内容、联系人、日程……这些最敏感的数据，要先上传到云端才能被"理解"。
- **延迟与成本**：每一次对话都要走网络，长上下文的 token 费用也不便宜。
- **可用性**：没网就不能用，弱网体验差。

端侧推理（On-device Inference）给出了另一条路：**模型直接跑在手机上，数据不出端**。随着 LiteRT-LM 等端侧推理框架的成熟，今天已经可以在手机上运行十亿级参数的多模态模型，做本地对话、图像理解、语义检索和动作规划。

**栖知 Solin**（Solin Android）就是这样一个实验性的、隐私优先的端侧 Agent。它的核心思路是：**能用本地模型解决的，就不上云；必须上云的，让用户知情并确认**。

## 二、Solin 里的端侧模型能力

Solin 在手机上跑三类模型，各司其职：

### 1. 聊天模型：LiteRT-LM（Text + Vision）

| 模型 | 大小 | 能力 |
| --- | --- | --- |
| chat-e2b（Gemma-4 E2B） | ~2.6 GB | 基础本地对话，支持文本 + 图像输入 |
| chat-e4b（Gemma-4 E4B） | ~3.7 GB | 更高质量的本地对话选项 |

- 基于 **LiteRT-LM** 框架，支持 GPU / CPU 自动回退。
- 支持 **Text + Vision**：可以直接理解用户在手机上分享的图片，不需要先传到云端 OCR。
- 本地图片输入有边界保护：最多 5 张、单张不超过 8 MiB，防止内存爆炸。
- 同时兼任**动作规划模型**：观察到设备状态后，规划下一步该点哪个按钮、输入什么文字。

### 2. 记忆模型：EmbeddingGemma 300M

| 模型 | 大小 | 能力 |
| --- | --- | --- |
| EmbeddingGemma 300M | ~179 MB | 把文本转成语义向量，做本地记忆检索 |

- 不是聊天模型，专门做 **Embedding**。
- 把对话历史、笔记、上下文存成向量，用户提问时做**语义相似度检索**，把相关内容喂给聊天模型。
- 记忆索引完全在本地建立和查询，**不上云**。
- 激活条件严格：主模型 + 分词器（sentencepiece）都通过 SHA-256 校验、运行时探测通过，才会启用语义记忆。

### 3. 动作模型：mobile-action-270m（实验性）

| 模型 | 大小 | 能力 |
| --- | --- | --- |
| mobile-action-270m | ~284 MB | 低资源的设备动作规划 |

- 当没有可用的聊天模型做规划时，作为**轻量 fallback**。
- 输入是观察到的设备状态（当前页面文本、可点击元素等），输出是下一步动作（点击、输入、滚动……）。
- 如果模型加载失败或输出无效，自动回退到**保守的规则规划**，不会瞎操作。

### 模型能力画像

每个模型都有结构化的**能力画像**（capability profile），声明它能做什么、不能做什么：

- `chat` / `vision` / `memoryEmbedding` / `mobileAction` 四类能力开关
- 输入模态（Text / Vision）
- 上下文窗口大小（如 chat-e2b 为 8192 tokens）
- 隐私属性：所有本地模型 `remoteEligible = false`，即它们处理的内容默认不可上传

这套画像不是装饰——它是后续路由、安全、隐私决策的基础。

## 三、典型应用场景

### 场景一：离线隐私聊天

- 用户在地铁上、没网时，照样可以和 AI 对话。
- 本地图片输入、OCR 结果等敏感内容会被显式标记为 `LocalOnly`，**永远不会自动上传**。
- 普通新建消息的领域模型默认值是 `RemoteEligible`；进入隐私敏感路径时必须显式收紧为 `LocalOnly`。
- 即使用户后来切换到远程模式，`LocalOnly` 的历史消息也会被过滤掉，不会发给云端。

### 场景二：本地记忆

- 用户可以和 AI 聊"上周我和张三讨论的那个方案"，AI 通过本地语义检索找到相关上下文。
- 记忆的建立、存储、检索全在端侧完成，**不依赖任何云端向量服务**。
- 用户可以随时清除记忆，应用卸载后数据随之消失。

### 场景三：模型驱动的应用搜索与低风险控制

这是端侧模型最有想象力的场景——**让 AI 理解屏幕并操作手机**：

- **应用搜索**：用户说"帮我打开点外卖的 app"，模型根据应用名称、描述做语义匹配，找到目标应用。
- **低风险控制**：观察当前屏幕 → 规划动作 → 执行（点击、输入、滚动、返回、等待）。
- **安全边界**：只允许低风险操作（observe / tap / type / scroll / back / wait）；发送、删除、支付、下单、权限修改等高风险操作**必须用户手动确认**。
- 模型只看到 Accessibility 树的结构化文本，**不直接接触原始屏幕截图**。

### 场景四：端云协同（Auto 模式）

不是所有任务都适合端侧——复杂推理、长上下文生成，本地模型可能太慢或太热。Solin 的 **Auto 模式**让手机自动决定：这个请求该本地跑，还是该上云？

决策顺序（**纯本地、确定性策略，不把原始 prompt 发给路由器**）：

1. **隐私边界**：内容是 `LocalOnly` 还是 `RemoteEligible`？前者只能本地。
2. **能力需求**：请求需要什么能力（视觉？长上下文？），本地模型是否支持？
3. **候选健康**：远程端点是否已配置、已授权、可达？
4. **资源与复杂度**：
   - 图片请求且本地支持视觉 → **优先本地**（图片不出端）
   - 低内存、设备过热（thermal）→ **选远程**
   - 复杂任务（多步计划、工具循环、4096+ 输出 token）→ **选远程**
   - 简单请求 → **选本地**

> 当前状态：Auto 模式在 debug 构建中以 `opt-in` 开放，release 变体默认关闭。真机验收和 release trace 尚未完成。

## 四、技术架构概览

Solin 的请求主路径可以概括为：

```
用户输入 → Skill-first 路由 → 本地模型 / 远程 / 工具
工具 → ToolRegistry 校验 → SafetyPolicy → 用户确认 → 执行 → 脱敏审计
```

几个关键设计：

### Agent Loop

- **Skill-first 预路由**：先判断请求是否匹配某个内置 Skill（如提醒、分享、导航），匹配则走工具路径，否则走模型对话。
- **观察-决策循环**：工具执行后，模型观察结果，决定下一步是继续调用工具、还是直接回答用户。
- **本地续写 vs 远程续写**：工具结果如果是 `LocalOnly`（如联系人、日历、屏幕文本），只能用本地模型续写；如果是公开证据（如网页搜索结果），可以用远程模型。

### 工具注册表（ToolRegistry）

- 每个设备能力都被描述为一个 `ToolSpec`，声明：名称、参数 schema、所需权限、风险等级、确认策略、隐私分类。
- 注册表在启动时**冻结**（freeze），拒绝重复注册、拒绝无 spec 的 handler、拒绝运行时动态修改——防止 Agent 循环和执行边界各持一份白名单。
- 输入输出都有 **schema 校验**：未知字段、缺失必填、类型错误、数值越界都会被拒绝。

### 安全与隐私边界

- **隐私值显式传播**：普通新建消息默认是 `RemoteEligible`；剪贴板、OCR、Accessibility 屏幕文本、分享摘录和本地动作等敏感路径会显式收紧为 `LocalOnly`，远程发送时再做 fail-closed 过滤。
- **确认式执行**：中风险及以上的工具调用，必须经过用户确认才能执行。
- **脱敏审计**：所有工具调用都有审计记录，但远程审计**不存储原始 prompt**，只保留发送决策、模型名、敏感类别、图片数、远程历史数量和脱敏摘要。
- **端侧加密**：设备上的证据 blob 使用 AES/GCM/NoPadding 认证加密，密钥存在 AndroidKeyStore 中；旧版明文和 AES/CBC blob 仅作为兼容读取并迁移。

## 五、当前状态与限制

Solin 目前处于**实验性 / 个人评估**阶段，适合本地开发和受控测试，**尚未 ready  for 应用商店或生产分发**。

主要限制：

| 维度 | 现状 |
| --- | --- |
| 模型许可 | 推荐模型为第三方产物，许可证、分发权、归属声明需人工逐一审核 |
| 真机验收 | Auto 模式、设备控制等核心能力尚未完成真机 instrumentation 验证 |
| Release 就绪 | release-like 构建的 Auto 模式默认关闭，签名、性能基线、法务审批均未完成 |
| JDK 要求 | 完整验证脚本需要 JDK 21 环境 |
| 模型分发 | 模型权重不进 Git，普通 release 不打包模型；内部 bundledModels 包仅限实验室验证 |

## 六、总结与展望

### 端侧模型的价值

- **隐私**：敏感数据不出端，这是端侧推理最核心的价值。
- **低延迟**：本地推理省去网络往返，简单请求响应更快。
- **离线可用**：没网也能用，弱网体验稳定。
- **成本**：本地推理没有 token 费用，适合高频、轻量的请求。

### 端侧模型的边界

- **能力上限**：手机能跑的模型参数量有限，复杂推理仍需云端。
- **资源约束**：内存、功耗、散热是硬约束，长时间高负载会触发 thermal throttling。
- **模型分发**：模型体积大（2-4 GB），如何分发、更新、校验是工程挑战。

### Solin 的下一步

- 完成 Auto 模式的真机验收和 release trace。
- 推进模型许可审核，明确分发边界。
- 完善设备控制的安全边界和用户确认流程。
- 优化端侧推理性能和资源管理策略。
